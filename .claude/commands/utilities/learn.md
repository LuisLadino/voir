# /learn - Explain Like I'm 5

Teach me anything. Code, science, history, philosophy, how engines work, why the sky is blue. Any topic.

```
/learn                     # Explain what we're currently working on
/learn [topic or question] # Explain anything
```

---

## This is NOT code-specific

This command explains ANY topic the user asks about. Do not refuse because something "isn't code related." If the user asks about quantum physics, medieval history, or how bread rises, explain it.

---

## How to Respond

### If no topic given
Explain what we're currently working on. Cover what it is, how it works, and why it's done that way.

### If topic given
Explain it like I'm 5 (but not patronizing):

1. **What it is** - One sentence, no jargon
2. **Why it matters** - The problem it solves or why anyone cares
3. **How it works** - Simple breakdown with analogy
4. **Example** - Concrete illustration from everyday life

Skip sections that don't apply. A question about history doesn't need "why it matters" if that's obvious.

---

## Principles

- **No jargon** - Define technical terms when first used
- **Use analogies** - Connect to things everyone knows
- **Be concrete** - Examples over abstractions
- **Adjust depth** - Simple first, add detail if asked
- **Be honest** - Say when something is genuinely complex

---

## Examples

**User:** `/learn why is the sky blue`
→ Explain light scattering using everyday analogies

**User:** `/learn how do vaccines work`
→ Explain immune response simply

**User:** `/learn kubernetes`
→ Explain container orchestration without assuming DevOps knowledge

**User:** `/learn the french revolution`
→ Explain what happened, why, and why it mattered

**User:** `/learn react hooks`
→ Explain what hooks are, why they exist, how to use them

---

## Tone

Clear, not condescending. Don't say "it's simple" or "obviously." If something is hard, acknowledge it and break it down further.

End by inviting follow-up questions.
